\chapter*{Executive Summary}
\addcontentsline{toc}{chapter}{Executive Summary}
\markboth{Executive Summary}{}

SIVO is a mobile application designed to facilitate real-time, two-way communication between deaf and hearing individuals in Pakistan. The application addresses a significant gap in the market by providing the first dedicated Pakistani Sign Language (PSL) translator.

The system operates in two modes: Sign-to-Speech, which converts hand gestures captured through the mobile camera into spoken audio using artificial intelligence, and Speech-to-Sign, which transforms spoken words into animated sign language performed by a 3D avatar.

Key features of SIVO include real-time translation, grammar correction using the Grok API, conversation history management, and a freemium monetization model. The application is built using React Native for the mobile frontend and Python Flask for the backend AI processing.

The primary target users are business owners and employees in organizations who need to communicate with deaf colleagues or customers. By eliminating communication barriers, SIVO aims to promote workplace inclusion and improve professional interactions for the deaf community.

This report documents the complete development lifecycle of SIVO, including requirements analysis, system design, implementation, and deployment procedures.

\chapter{Introduction}

In Pakistan, many people are deaf and silent, and they face many challenges that limit their ability to communicate effectively. First, in the environment of business and workplace, deaf persons struggle to engage in meetings, distribute presentations, or serve customers efficiently, which affects their professional development and confidence. Secondly, listeners also find it difficult to interact naturally with their deaf colleagues or customers, leading to social and professional isolation for both sides. Third, there is still no reliable and accessible digital solution that can translate specifically PSL sign language into speech in real time. There are several applications and websites for languages like ASL and BSL but not for local PSL.

The project, called SIVO, is designed to solve this problem. SIVO is a mobile application that works like Google Translator, but instead of converting a spoken language into another, it turns Pakistani Sign Language (PSL) into speech, and turns speech back into PSL. For example, when a deaf person signs in front of the camera, the application will turn it into clear Urdu or English speech. And when a hearing person speaks, the application will turn it into PSL gestures on the screen. In this way, both sides can eventually understand each other without the requirement of a third person.

To make it possible, a dataset of normal PSL signals will first be collected. Then, a machine learning model will be trained to correctly identify these signs. After that, a mobile application will be created that adds camera to sign input and uses speech technology for voice translation. Finally, the application will be tested with business people and employees to ensure that it works in real situations. Step by step, SIVO will transform from an idea into an actual tool that helps thousands of people to communicate better in their jobs, businesses, and everyday life.

\section{Project Introduction}

“The goal of this project was to develop SIVO, a mobile application capable of bidirectional translation between Pakistani Sign Language (PSL) and spoken language”.

To build this system, a diverse PSL dataset was collected, incorporating pre-existing legacy recordings, user webcam captures, and 1080p mobile videos across more than 140 word classes. Rather than training on raw images using Convolutional Neural Networks (CNNs), the system utilized MediaPipe Holistic to extract dynamic facial, pose, and hand landmarks. This sequential data was preprocessed using a sliding window technique and extensive data augmentation to train a robust GRU-based Recurrent Neural Network capable of recognizing temporal, continuous signs. The final model was optimized using TensorFlow Lite (TFLite) for high-speed inference.


Following word prediction, a custom "Smart Match" algorithm was implemented to map recognized words into grammatically appropriate sentences based on a conversational knowledge base (with third-party APIs available as a fallback). For the reverse communication flow (Speech-to-Sign), on-device speech recognition transcribes spoken language. An NLP pipeline lemmatizes the text and maps the resulting words to a bundled dictionary of pre-recorded 3D avatar videos, displaying them as a seamless visual sequence.
All of these phases—data collection, model training, prediction, sentence formation, and speech conversion—have been successfully integrated into a real-time, cross-platform mobile application. 


A deaf user can sign in front of the camera, and a hearing user can simply speak, enabling smooth two-way communication. The primary beneficiaries of this project are businesses and organizations in Pakistan, where communication barriers with deaf individuals have historically hindered productivity and workplace inclusion \hyperlink{ref1}{[1]}, \hyperlink{ref7}{[7]}.



\textbf{Business People:} SIVO helps business owners and entrepreneurs communicate smoothly with clients, employees, and partners. It removes the communication barrier during meetings, customer interactions, and daily operations, making business activities more inclusive and efficient.

\textbf{Employees in Large Organizations:} SIVO supports employees, especially those in corporate or office environments, by enabling clear two-way communication between deaf and hearing staff. It ensures better teamwork, understanding in meetings, and overall workplace accessibility.

\subsection{System Design Document}

\textbf{a) App Concept}

The main concept is to create a two-way, real-time communication bridge.

\begin{itemize}
  \item \textbf{Sign-to-Speech:} The system uses machine learning, specifically MediaPipe and feed-forward network (MLP) model, to analyze the user's hand, currency, and facial gestures through a live video feed, converting them into Urdu or English.
  \item \textbf{Speech-to-Sign:} To display the correct PSL signals on the screen, a custom animation/gesture engine uses speech-to-text processing, which translates the spoken word into visual communication.
\end{itemize}

The creation of the application includes PSL gestures, advanced AI model training, and a simple, user-friendly mobile interface with comprehensive data collection.

\textbf{b) Target Audience}

\begin{itemize}
  \item \textbf{Primary:} Business owners and employees in big organizations who are deaf and mute individuals who use Pakistani Sign Language (PSL).
  \item \textbf{Secondary:} Family members, teachers, businesses, healthcare professionals, and anyone who regularly interacts with the deaf community.
\end{itemize}

  \end{itemize}
\end{itemize}

\textbf{d) Project Scope}

The scope of this project is to design and develop SIVO, a mobile application that bridges the communication gap between the deaf/mute community and the speaking population. The application will provide real-time translation of Pakistani Sign Language (PSL) into spoken Urdu/English and convert spoken Urdu/English into PSL gestures using animations or visuals.

\textbf{Inclusions:}
\begin{itemize}
  \item Real-time sign recognition through the mobile camera.
  \item Conversion of PSL gestures into speech (Urdu and English) (300 to 400 words).
  \item Conversion of spoken Urdu/English into PSL animations.
  \item A simple, user-friendly interface accessible to all age groups.
  \item Support for mobile platforms (Android initially, iOS later if time allows).
\end{itemize}

\textbf{Exclusions:}
\begin{itemize}
  \item Support for international sign languages beyond PSL (e.g., ASL, BSL).
  \item Advanced personalization features (like saving conversations, offline training).
  \item No more than 400 words, but the dictionary can be increased with time.
\end{itemize}

\textbf{Boundaries:} This project will focus only on basic daily-life conversations and business-related communication between deaf/mute users and normal speakers within Pakistan. The solution will be scalable in the future to add more languages, advanced features, and cross-platform support.

\subsection{Monetization Model}

Since this project is based on a mobile application that translates Pakistani Sign Language (PSL) into text/speech, several ways can be explored to make it sustainable and profitable:

\begin{itemize}
  \item \textbf{Freemium (Main Model):} The application will be free to use for basic translations (daily conversation gestures). Users who need advanced features, like business vocabulary packs, offline usage, or premium voices, can upgrade to the paid version.
  \item \textbf{Advertising (Optional):} Light, non-intrusive ads can be shown in the free version (for example: small banners at the bottom). But since this application serves a sensitive audience, ads should never interrupt or annoy the user.
\end{itemize}

\section{Useful Tools and Technologies}

\textbf{1. Programming Language}
\begin{itemize}
  \item Python will be used for building the sign recognition model because it is the best choice for AI/ML projects. It has powerful libraries like TensorFlow, Keras, and OpenCV which make gesture recognition easier.
  \item For the mobile application itself, Kotlin will be used because it allows building Android applications.
\end{itemize}

\textbf{2. Development Environment}
\begin{itemize}
  \item PyCharm / Jupyter Notebook for building and testing the AI model (because they are great for debugging and experimenting with ML code).
  \item Android Studio / VS Code for Kotlin application development (because they provide strong support for mobile UI design and debugging).
\end{itemize}

\textbf{3. Database}
\begin{itemize}
  \item Firebase will be used because it is cloud-based, real-time, and integrates smoothly with Flutter applications. It will store user data, application preferences, and sign vocabulary sets.
\end{itemize}

\textbf{4. Operating Systems}
\begin{itemize}
  \item Android is the main target since the application will be mobile-first.
  \item Later, the project can also be extended to Web for schools and offices.
\end{itemize}

\textbf{5. Network Protocol}
\begin{itemize}
  \item HTTP/HTTPS will be used to connect the mobile application with cloud services (Firebase, APIs, or AI model updates).
  \item This ensures secure communication, especially when sending/receiving sign translation data.
\end{itemize}

\chapter{Requirement Specification and Analysis}

This section provides a comprehensive explanation of the complete functionality of the SIVO application. It encompasses the major features, user stories, test cases, and the operational workflow of the entire system including sign-to-speech conversion, speech-to-sign translation, grammar correction mechanisms, user interface design, performance optimization, and security implementations. Each epic represents a group of related features, while each user story describes a specific user requirement. Every story is accompanied by a test case to verify its correct functionality.

\section{SIVO Epics}

\subsection{Epic 1: User Account and Security}

\textbf{Description:}

As a user, the requirement is to create and manage a secure personal account so that conversation history, custom settings, and personal data remain private and accessible only to the authorized user. This epic encompasses the entire authentication process. Since SIVO handles personal conversations between deaf and hearing users, security is considered a top priority. This module ensures that users can sign up easily, log in quickly using modern methods such as Google authentication, and safely log out upon completion, ensuring that no unauthorized person can access their private translations.

\subsection{Epic 2: Sign-to-Speech Translation}

\textbf{Description:}

As a deaf user, the requirement is to utilize the mobile camera to record hand gestures and have them converted into spoken audio so that communication with hearing people becomes possible without needing a human interpreter. This epic represents the core feature for deaf users. It involves capturing video, utilizing Artificial Intelligence to recognize hand shapes, correcting the grammar of the detected words to make them sound natural, and finally playing the result out loud. The objective is to bridge the gap from visual language to audio language in real-time.

\subsection{Epic 3: Speech-to-Sign Avatar}

\textbf{Description:}

As a hearing user, the requirement is to speak into the application and observe a 3D avatar perform the corresponding sign language movements so that messages can be conveyed to a deaf person visually. This epic represents the core feature for hearing users. It handles the reverse process: capturing voice, converting it to text, finding the correct 3D animation for each word, and rendering a smooth 3D character on the screen. This enables the deaf user to visually perceive what is being said.

\subsection{Epic 4: Conversation History and Chat}

\textbf{Description:}

As a user, the requirement is to save translations in a chat-like history log so that past conversations can be reviewed and important information exchanged during the session can be tracked. This epic manages the data persistence of the application. It ensures that the translation is not merely a one-time event but is recorded in a ``Live Chat'' screen while it occurs, and then permanently archived in a ``History'' tab. This makes the application feel like a complete communication tool, similar to WhatsApp, rather than just a simple translator.

\subsection{Epic 5: Monetization and Freemium Model}

\textbf{Description:}

As a product owner, the requirement is to implement a Freemium business model with advertisements and premium plans so that the application generates revenue to cover server costs while remaining accessible to free users. This epic ensures the sustainability of the project. It defines how free users interact with the application through viewing advertisements and what benefits paid users receive such as no advertisements and faster processing. This balance is crucial for maintaining the application's long-term operation.

User Stories are concise, user-focused descriptions of specific features derived from epics. These represent small, actionable requirements that can be completed within a short development cycle. Each user story captures what the user needs and the rationale behind it, ensuring the system is developed from the end user's perspective.

  \item \textbf{Microphone:} Built-in microphone with noise cancellation support.
  \item \textbf{Storage:} 200 MB of free internal storage for app installation and temporary video cache.
  \item \textbf{Internet Connectivity:} Stable Wi-Fi or 4G LTE connection (Minimum 1 Mbps upload speed for sending video data to the server).
\end{itemize}

\section{Limitation and Constraint}

This section lists the technical and environmental limitations of the SIVO application.

\begin{itemize}
  \item \textbf{Active Internet Required:} The application requires a continuous and stable internet connection to communicate with the AI server; offline translation is not supported.
  \item \textbf{Lighting Conditions:} The Sign-to-Speech module requires a well-lit environment for the camera to accurately detect hand gestures and facial expressions.
  \item \textbf{Single User Focus:} The current version is optimized to detect signs from a single person within the camera frame; multiple users signing simultaneously may cause detection errors.
  \item \textbf{Camera Positioning:} The user must position their mobile device at a distance of approximately 1 to 2 meters to ensure their upper body (hands and face) is fully visible.
  \item \textbf{Language Support:} The Speech-to-Sign module currently supports English language input only.
\end{itemize}

\section{Software Architecture}

For SIVO, a Cloud-Backed Client-Server Architecture was utilized. This hybrid approach ensures that heavy computational tasks (such as artificial intelligence inference and NLP processing) occur on a powerful cloud server, while static media assets remain local to the device to guarantee zero-latency playback.

The major architectural decisions guiding the SIVO system are:
\begin{itemize}
  \item \textbf{Application Type:} A mobile application frontend supported by an intelligent cloud backend.
  \item \textbf{Deployment Strategy:} The mobile client runs natively on user devices (Android/iOS), while the AI backend runs on cloud infrastructure.
  \item \textbf{Appropriate Technologies:} React Native (Expo) for the mobile interface, Python (Flask) for the AI/NLP server logic, Firebase for user authentication, and a NoSQL hybrid approach (MongoDB + Local Context) for data storage.
  \item \textbf{Quality Attributes:} Low latency, offline robustness for video rendering, and high linguistic accuracy.
\end{itemize}

The system is logically divided into three main layers:

\begin{enumerate}
  \item \textbf{Mobile Client (Frontend):} Built with React Native. This layer handles all user interactions. It utilizes the device camera for sign capture, native speech-to-text engines for voice capture, and a custom-built video sequencer to seamlessly play pre-rendered sign language assets back-to-back without buffering.
  \item \textbf{Intelligent Backend (Server):} A Python Flask server that acts as the ``Brain'' of SIVO. It comprises three sub-engines:
  \begin{itemize}
    \item \textbf{Sign Recognition Engine:} Uses MediaPipe and a GRU/LSTM model to track hand landmarks and predict raw sign language words.
    \item \textbf{Grammar Engine:} Utilizes an LLM API to format broken, raw sign predictions into grammatically correct English sentences.
    \item \textbf{NLP Pipeline (Speech-to-Sign):} Uses the spaCy library to process English speech, strip out stop words (e.g., ``is,'' ``the,'' ``am''), and lemmatize verbs (e.g., converting ``running'' to ``run'') to map spoken language accurately to available Pakistan Sign Language (PSL) root words.
  \end{itemize}
  \item \textbf{Data Layer:} Firebase Authentication securely handles user identity and login sessions. MongoDB (Atlas) serves as the primary NoSQL database for dynamic application data (user profiles and chat histories). Simultaneously, a local JSON-based VideoDictionary is bundled with the application to map processed text directly to local video assets.
\end{enumerate}

\section{Data Modeling}

SIVO utilizes a document-based NoSQL architecture (MongoDB) for persistent cloud storage and lightweight local data structures for high-performance mobile execution. Because SIVO functions as a communication tool, the database is structured similarly to modern messaging applications (e.g., WhatsApp), separating users, conversation threads, and individual messages.

\subsection{NoSQL Databases (MongoDB and Local Structures)}

The data model relies on the following distinct collections and local structures to manage the application's core logic:

\textbf{1. Users Collection:}
While Firebase Authentication securely handles user credentials and login tokens, this MongoDB collection stores supplementary personal account details and application preferences. It utilizes the Firebase User ID (UID) as the primary document key to link the database to the authenticated user.

\begin{lstlisting}[caption={Users Collection JSON Structure},captionpos=b]
{
  "_id": "firebase_uid_10001",
  "full_name": "Muhammad Abubakar",
  "email": "abubakar@example.com",
  "auth_provider": "google",
  "subscription_tier": "freemium",
  "preferences": {
    "dark_mode": true,
    "voice_speed": 1.0
  },
  "created_at": "2026-04-20T10:00:00Z",
  "last_login": "2026-04-25T14:30:00Z"
}
\end{lstlisting}

\textbf{2. Conversations Collection:}
Acts as the parent container for different chat sessions. This allows users to have distinct, separated conversation threads depending on who they are speaking with, mimicking a standard chat interface.

\begin{lstlisting}[caption={Conversations Collection JSON Structure},captionpos=b]
{
  "_id": "C50012",
  "owner_id": "firebase_uid_10001",

  "updated_at": "2026-04-25T15:05:00Z",
  "created_at": "2026-04-25T14:45:00Z"
}
\end{lstlisting}

\textbf{3. Messages Collection:}
Stores the individual translation events linked to a specific conversation thread. It records whether the message was generated via sign language or voice, alongside the raw and corrected outputs.

\begin{lstlisting}[caption={Messages Collection JSON Structure},captionpos=b]
{
  "_id": "M90045",
  "conversation_id": "C50012",
  "sender_id": "firebase_uid_10001",
  "translation_direction": "sign_to_speech",

  },
  "timestamp": "2026-04-25T14:50:12Z"

\end{lstlisting}

\textbf{4. VideoDictionary Mapping (Local Data):}
Instead of storing heavy 3D coordinates in the database, the Speech-to-Sign module uses a localized key-value JSON structure bundled directly inside the React Native application. It acts as an instant lookup table connecting lemmatized words to local video assets.

\end{lstlisting}

\subsection{Epic 1: User Account and Security}
This workflow manages user access and data privacy.

  \item The user opens the application and selects a login method 
  \item Credentials are sent securely to the backend for verification.
  \item Upon success, a secure session token is generated and stored locally on the device.
  \item User data is retrieved from the MongoDB \texttt{Users} collection, granting access to the main dashboard.
\end{enumerate}

\subsection{Epic 2: Sign-to-Speech Translation}
This workflow demonstrates how a Deaf user's hand gestures become spoken words for a hearing person.
\begin{enumerate}
  \item \textbf{Capture:} The user records a video of their sign language performance.
  \item \textbf{Upload:} The video payload is transmitted to the Flask Server.
  \item \textbf{Extraction:} MediaPipe extracts spatial hand and body coordinates from the frames.
  \item \textbf{Prediction:} The AI model (LSTM/GRU) analyzes the data to output raw keywords (e.g., ``Me Go Store'').
  \item \textbf{Correction:} The Grok API reformats the raw words into a grammatically correct sentence (e.g., ``I am going to the store'').
  \item \textbf{Output:} The text is returned to the phone and vocalized via a Text-to-Speech engine.
\end{enumerate}

\begin{figure}[H]
\centering

\label{fig:wf-epic2}
\end{figure}

\subsection{Epic 3: Speech-to-Sign Avatar}
This workflow details how spoken language is seamlessly translated into a visual sign performance for a Deaf user.
\begin{enumerate}
  \item \textbf{Input:} The hearing user speaks into the application's microphone.
  \item \textbf{Transcription:} The native react-native-voice engine converts the spoken audio into text locally.
  \item \textbf{NLP Pipeline:} The Flask server uses the spaCy library to tokenize the sentence, strip English stop words, and lemmatize verbs.
  \item \textbf{Array Generation:} The server returns a JSON array of validated root words (e.g., \texttt{["i", "go", "office"]}).
  \item \textbf{Sequencing \& Rendering:} The expo-av module looks up the array in the local VideoDictionary and plays the video clips sequentially, triggering the next video the exact millisecond the previous one finishes.
\end{enumerate}

\subsection{Epic 4: Conversation History and Chat}
This workflow explains how translations are persisted to mimic a continuous messaging application.

\chapter{Software Development}

The Software Development chapter describes how the SIVO system was realized at a finer level of detail, down to the code level. While the previous chapters focused on planning and architecture, this section details the coding process, the logic behind the features, and the deployment of the development environment. It ensures that the system design presented in Chapter 3 directly corresponds to the final implementation.

During the implementation phase, several unforeseen problems were encountered, and a significant amount of project time was spent overcoming them through architectural pivots:

\begin{itemize}
  \item \textbf{Challenge 1: Real-Time 3D Avatar Rendering on Mobile:} Initially, the project aimed to calculate inverse kinematics and render 60+ skeletal bones at 60 FPS in real-time on the user's mobile device. \textbf{Resolution:} This proved overambitious and caused severe battery drain, frame drops, and application crashes on mid-tier devices. To overcome this, the implementation was pivoted to a \textit{Pre-Rendered Video Sequencer}. 3D avatar animations were rendered into highly compressed local video assets, and the React Native frontend was coded to sequence these videos instantly back-to-back, guaranteeing zero latency without overloading the mobile GPU.
  \item \textbf{Challenge 2: Continuous Sign Prediction and Temporal Modeling:} Training an AI model to predict continuous signs from a single camera feed resulted in poor accuracy when trying to process the entire video at once, as the model struggled to separate individual signs in a long sequence. \textbf{Resolution:} The approach was modified to use a Sliding Window technique combined with a Gated Recurrent Unit (GRU) network. Instead of processing a 190-frame video as a single block, a fixed-size window of 30 frames slides across the sequence frame-by-frame (e.g., frames 1-30, 2-31, 3-32). The GRU model, trained specifically on these 30-frame temporal windows, processes each segment to predict the sequence of signs accurately before passing them to the external LLM (Grok API) for grammar correction.
\end{itemize}

\section{Coding Standards}

To maintain a clean, professional, and easily maintainable codebase across a distributed architecture, strict coding standards were enforced throughout the development process.

\begin{itemize}
  \item \textbf{Indentation:} A strict 4-space indentation rule was applied to the Python (Flask/AI) backend to prevent \texttt{IndentationError} exceptions. For the React Native (JavaScript) frontend, a 2-space indentation standard was used to prevent excessive horizontal scrolling in deeply nested UI component trees.
  \item \textbf{Naming Conventions:} 
    \begin{itemize}
      \item \textit{Variables and Functions:} \texttt{camelCase} was used for JavaScript frontend components (e.g., \texttt{fetchSequence}, \texttt{startRecording}), while \texttt{snake\_case} was used for Python backend logic (e.g., \texttt{extract\_landmarks}, \texttt{process\_video}).
      \item \textit{Classes and Components:} \texttt{PascalCase} was strictly used for React components and Python classes (e.g., \texttt{SpeechToSignScreen}, \texttt{SignDetector}).
      \item \textit{Constants:} \texttt{UPPER\_SNAKE\_CASE} was used for global constants and environment variables across all environments (e.g., \texttt{API\_BASE\_URL}, \texttt{WINDOW\_SIZE}).
    \end{itemize}
  \item \textbf{Statement Standard:} Code was modularized into single-responsibility functions. Complex AI matrix calculations and asynchronous API calls were wrapped in \texttt{try/catch} (JS) or \texttt{try/except} (Python) blocks to prevent fatal application crashes during network instability.
\end{itemize}

def predict_with_sliding_window(video_frames, model, window_size=30):
    predictions = []
    total_frames = len(video_frames)
    
    # Slide a window of 30 frames across the entire video (e.g. 1-30, 2-31...)
    for i in range(total_frames - window_size + 1):
        window = video_frames[i : i + window_size]
        
        # Expand dimensions to match GRU expected input: (batch, timesteps, features)
        input_data = np.expand_dims(window, axis=0)
        
        # Pass the 30-frame temporal window to the trained GRU model
        pred = model.predict(input_data)[0]
        predictions.append(pred)
        
    return aggregate_predictions(predictions)
\end{lstlisting}
\textbf{Description:} This function is responsible for the temporal processing of the sign language video. Instead of processing a full variable-length video (e.g., 190 frames) at once, it applies a sliding window approach to maintain consistent input shape for the model. The video frames are divided into overlapping sequences of 30 frames, which matches the input requirement of the trained GRU model. Each sequence is independently analyzed to predict the probability of a sign, and the outputs from all windows are aggregated to generate the final continuous word-level transcription.

\section{Software Limitations and Constraints}

While SIVO is designed to be robust, there are certain limitations and constraints in the current version:

\begin{itemize}
  \item \textbf{Active Internet Connection:} The mobile application requires a stable internet connection to communicate with the Flask server and Grok API. It cannot perform translations offline in this version.

\end{itemize}

  \item \textbf{Database User Configuration:} Create a secure Database User with read/write privileges and generate a secure connection string URI.
\end{enumerate}

\subsection{Phase 3: Backend Server Deployment (Microsoft Azure)}

The Python Flask server, which houses the GRU sliding-window sign recognition and Grok NLP logic, is deployed to Microsoft Azure using the Azure App Service.

\begin{enumerate}
  \item \textbf{Create Azure App Service:} Navigate to the Microsoft Azure Portal. Search for "App Services" and click "Create." Select a Linux environment and choose Python 3.9 (or higher) as the runtime stack.
  \item \textbf{Configure Environment Variables:} Navigate to the "Configuration" pane within the App Service. Here, add the critical application secrets that should not be hardcoded in the repository:
  \begin{itemize}
    \item \texttt{MONGO\_URI}: The connection string generated in Phase 2.
    \item \texttt{GROK\_API\_KEY}: The secret key for accessing the LLM.
  \end{itemize}
  \item \textbf{Deploying the Code:} Connect the Azure App Service directly to the project's GitHub repository. Configure GitHub Actions to automatically trigger a build and deployment whenever new code is pushed to the \texttt{main} branch. Azure automatically reads the \texttt{requirements.txt} file and installs necessary dependencies like \texttt{flask}, \texttt{spacy}, \texttt{tensorflow}, and \texttt{mediapipe}.
  \item \textbf{Start and Verify:} Once deployed, Azure provides a live URL (e.g., \texttt{https://sivo-backend.azurewebsites.net}). Navigate to the root endpoint to verify the server is running.
\end{enumerate}

\subsection{Phase 4: Mobile Client Deployment (React Native / Android)}

With the backend and database live, the mobile application must be configured with all live API keys and compiled into an installable format.

\begin{enumerate}
  \item \textbf{Environment Configuration:} Open the React Native project code. Update the \texttt{.env} file with the Firebase API keys from Phase 1, and change the \texttt{API\_BASE\_URL} from the local development IP to the live Azure server URL.
  \item \textbf{Asset Bundling:} Ensure all pre-rendered 3D avatar video assets (.mp4 files) are correctly linked in the \texttt{VideoDictionary.js} file so they are bundled directly into the application payload.
  \item \textbf{Building the APK:} Open the terminal in the project root directory. Use Expo Application Services (EAS) to compile the code into a standalone Android package by running the command: \texttt{eas build -p android --profile preview}.
  \item \textbf{Download and Install:} Once the build is complete on the Expo servers, download the generated \texttt{.apk} file. Transfer the file to an Android device, enable "Install from Unknown Sources" in the device security settings, and tap the APK to install the SIVO application.
\end{enumerate}



\section{Post-Deployment Scalability and Maintenance}

To ensure SIVO remains highly available as the user base grows, the Azure App Service was configured with auto-scaling rules. If CPU utilization exceeds 75\% due to heavy AI video processing requests and GRU model inference, Azure will automatically spin up additional instances to handle the load. Furthermore, because the 3D avatar video assets are bundled locally on the device rather than streamed over the network, the server bandwidth costs remain minimal, adhering to the project's resource-efficient design philosophy.
